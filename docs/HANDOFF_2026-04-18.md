### Phase 4.5 — Tiered offline architecture

**When to trigger:** product catalog crosses 50k entries, OR Flutter app bundle exceeds 60 MB. Not now — beta's 10k fits fine in the current ~32 MB bundle.

**Why this matters:** the scale trajectory is 10k → 50k → 250k (all DSLD). A naive 250k local DB would be ~575 MB — app-store-unshippable. Research during the session found Yuka (6M products) ships only 1.6% locally; we should adopt the same pattern.

**Where to find:**

- Full architecture + Yuka/MyFitnessPal/Cronometer research: `docs/AUTOMATION_ROADMAP.md` § "Phase 4.5 — Tiered Offline Architecture"
- Scale math (sqlite-zstd gives 6.5x compression; 250k × 150 bytes/row ≈ 15 MB): same section
- Pro-tier monetization layer: same section § "Monetization layer — Pro-tier offline gating"

**Architecture decisions already made:**

- **Three tiers:** Tier 1 (50k in app binary, ~3 MB) → Tier 2 (150k post-install pack via iOS ODR / Android Play Asset Delivery, ~15 MB) → Tier 3 (all 250k + blobs on Supabase).
- **Pro gating:** Free tier gets Tier 1 + online Tier 3. Pro adds Tier 2 offline pack.
- **Snapshot + delta sync** via HTTP Range requests (NOT Supabase Realtime for product data, NOT PowerSync subscription).
- **New pipeline stage `build_verdict_shard.py`** to emit the compressed shards.

## Phase 1 follow-up wins (add these when building Phase 1)

These are not-previously-deferred items that emerged from the tonight's ops work. Each is a real, verified gap, not a theoretical one. Bake them into Phase 1 construction rather than filing as separate future tickets.

### 1. Supabase sync `--cleanup` config (automated path)

**Problem:** `sync_to_supabase.py --cleanup` is an explicit CLI flag. If it's forgotten on an automated run, Supabase Storage accumulates ~17 MB of historical blobs per release. 52 weekly releases = ~900 MB/year of dead bytes.

**Fix:** Create `config/supabase_sync.json`:

```json
{
  "cleanup": true,
  "cleanup_keep": 5,
  "require_explicit_force": false
}
```

Update `sync_to_supabase.py` to read config if present, CLI flags override. Phase 1 `build-db.yml` workflow calls sync with no flags → config defaults → always cleans up with a 5-release rollback window (one work-week).

**Why keep=5 instead of default 2:** tight enough to bound Storage growth (~85 MB resident), generous enough to survive a 3-week-late bug discovery. Default-2 is too narrow for clinical-app reliability.

**Effort:** ~30 min during Phase 1 construction.

### 3. No PR-check CI workflow exists at all

**Problem:** `.github/workflows/` directory doesn't exist in either repo. The 6 production release-gate tests committed tonight (`test_safety_copy_production.py`) have no automated enforcement. A future PR that corrupts `scripts/data/*.json` would pass local testing (if the author skipped the prod tests) and merge before anyone notices.

**Fix:** create `.github/workflows/validate-pr.yml` in the pipeline repo as part of Phase 1:

```yaml
name: validate-pr
on: { pull_request: { paths: ['scripts/**', 'tests/**', 'config/**'] } }
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: '3.13' }
      - run: pip install -r requirements-dev.txt
      - run: python3 scripts/validate_safety_copy.py --strict
      - run: python3 -m pytest scripts/tests/ -q
```

Then in GitHub repo settings → branch protection → require `validate` status check before merge. **PRs can't merge with red.**

**Effort:** ~1 hour during Phase 1 construction. Single most foundational gap today.

### 4. Catalog health assertion on Supabase

**Problem:** right now nobody will notice if a bug in the pipeline silently ships a catalog with 50 products instead of 4,240. The sync gate checks `product_count >= 2000` but if someone bumps that threshold down during debugging, there's no second line of defense.

**Fix:** a tiny Supabase Edge Function that runs after every sync:

```typescript
// supabase/functions/catalog-health-check/index.ts
const { count } = await supabase.from('products_core').select('*', { count: 'exact', head: true });
if (count < 4000) {
  await postToSlack(`🚨 Catalog health degraded: ${count} products (expected ≥ 4000)`);
}
```

Triggered via Supabase database webhook on `export_manifest` table insert. Free tier, runs in <100ms.

**Effort:** ~1 hour during Phase 5 observability work (can also slot into Phase 1 if convenient).

### 5. Pipeline determinism for CI

**Problem:** `build_final_db.py` uses `datetime.now()` for the `db_version` field, so two CI runs on identical input produce different output (different version strings, different checksums). This makes it hard to diff "what actually changed in this release."

**Fix:** make `db_version` deterministic based on:
- Git commit SHA of the pipeline + data files
- OR a hash of sorted inputs

CI runs become reproducible; `git bisect` becomes possible on the artifact level; duplicate-release detection becomes trivial.

**Effort:** ~2 hours. Worth doing during Phase 1 observability work, not urgent.
